# AI 学习日志：2026-05-02

## 今日主题

围绕 AI 财务报销信息抽取练习项目，重点学习和实践了 Prompt 工程、结构化 JSON 输出约束、规则抽取、风险点识别，以及最小 RAG 审核流程。

## 学习成果

### 1. 明确了项目目标和接口边界

- 项目定位为 `Node.js + Express` 的 AI 财务报销信息抽取练习项目。
- 明确核心接口 `POST /extract-reimbursement`，从中文自然语言报销描述中抽取结构化字段。
- 输出字段固定为：
  - `报销类型`
  - `候选报销类型`
  - `金额`
  - `日期`
  - `风险点`
- 建立了“无法明确判断就返回 `null`，不要胡编”的基本原则。

### 2. 设计并优化了报销抽取 Prompt V2

- 从第一版规则中总结出不足，包括风险覆盖不完整、日期处理不明确、金额语义优先级不足、多类型候选处理不清晰等问题。
- 在 `docs/prompt-v2.md` 中形成了第二版 Prompt。
- Prompt V2 强化了以下约束：
  - 只输出合法 JSON，不输出 Markdown 或解释文字。
  - 不得输出 schema 之外的字段。
  - 报销类型、金额、日期必须来自文本明确证据。
  - 相对日期在没有上下文时返回 `null`。
  - 多金额、多类型候选、金额缺失、日期缺失等场景必须进入风险点。

### 3. 建立了测试样例驱动的规则迭代方式

- 在 `docs/test-cases.md` 中整理了财务报销抽取测试样例。
- 覆盖了差旅费、餐饮费、办公费、交通费、日期缺失、金额缺失、发票金额不一致、多类型候选、文本信息不足、异常金额等场景。
- 每个测试样例都包含：
  - 输入文本
  - 期望 JSON
  - 判断原因
- 通过测试样例把 Prompt 规则从“自然语言描述”落到了可验证的行为预期上。

### 4. 实现了本地 mock 抽取与模型调用回退思路

- 项目支持在没有模型 API Key 时使用本地 mock 逻辑，保证本地开发和测试不依赖外部服务。
- `/extract-reimbursement` 支持优先调用 Gemini；未配置 `GEMINI_API_KEY` 或模型调用失败时，可回退到本地 mock。
- 学到了一个实用工程原则：AI 功能也应该有稳定的本地 fallback，避免开发、测试和演示完全依赖远程模型。

### 5. 学习了结构化输出校验的重要性

- 对抽取结果增加响应结构校验，确保输出字段类型稳定。
- 对请求参数增加校验，例如：
  - `text` 必须是字符串。
  - `text` 不能为空。
- 对模型返回内容增加 JSON 解析校验，模型返回非法 JSON 时需要明确识别并降级处理。

### 6. 完成了最小 RAG 审核版本

- 新增本地制度文档 `knowledge/reimbursement-policy.md`，作为报销审核的知识来源。
- 实现 `POST /audit-reimbursement` 审核接口。
- 第一阶段不引入向量数据库，而是直接读取整份制度文档，再将制度内容和报销文本一起交给模型审核。
- 审核输出固定为：
  - `报销类型`
  - `金额`
  - `日期`
  - `风险点`
  - `制度依据`
  - `是否需要人工复核`
- 理解了最小 RAG 的核心闭环：本地知识文档 -> Prompt 注入 -> 模型审核 -> 结构化输出。

### 7. 用自动化测试固化关键行为

- 使用 Node.js 内置 `node:test` 为抽取和审核流程补充测试。
- 测试覆盖了：
  - 正常抽取
  - 字段缺失
  - 相对日期
  - 多金额风险
  - 多类型候选
  - 异常金额
  - 参数错误
  - 本地制度读取
  - 无 API Key 时 mock 回退
  - 审核接口固定 JSON 输出
- 认识到 Prompt 工程项目同样需要测试，尤其要用测试样例锁定边界行为。

## 今日关键理解

1. Prompt 不是单纯写提示词，而是在定义一套可执行、可验证的业务规则。
2. 结构化输出必须明确 schema、字段类型、无法判断时的返回策略和禁止项。
3. “不要胡编”需要具体化为 `null`、空数组、风险点等工程规则。
4. AI 抽取系统需要 fallback、校验和测试，不能只依赖模型一次性返回正确结果。
5. RAG 可以从最小版本开始，不必一开始就引入向量数据库；先把知识文档、Prompt 和结构化输出链路跑通更重要。
